# Inquix System Narrative Explanation

### Project Overview and Mission
Inquix is designed as a notebook-style knowledge base search assistant and research tool, mimicking the workspace flow of advanced reading hubs. The main objective of the platform is to allow users to establish knowledge bases, upload multi-modal research resources, and interface with them dynamically through a chat terminal. If the user's uploaded files do not contain information related to their question, the system falls back to active web search scraping to build context. The application features a decoupled architecture with a Python Django backend running under an asynchronous ASGI server, paired with a modern React frontend powered by Next.js. All components, including the relational vector database, cache servers, and local model inference layers, are managed together via Docker orchestration.

### In-Depth Document and Text Processing Pipeline
Document ingestion begins in [views.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/views.py) when a user uploads files. The backend utilizes [extraction.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/services/extraction.py) to read and extract data. Text-based files such as markdown, plain text, and HTML are parsed directly using standard Python file handles and string decoders. Microsoft Word files are processed by decompressing the docx archive structure using Python's standard `zipfile` module and parsing the paragraph XML tags with `xml.etree.ElementTree` to reconstruct the document outline without external library dependencies. Legacy doc files are processed through the `antiword` system utility invoked via a Python subprocess, falling back to an ASCII regex string scanner if the system tool is missing. PDF documents are ingested via the PyMuPDF library (imported as `fitz`), which retrieves layout and page metadata in a non-blocking stream. Audio attachments are sent through a subprocess that leverages the `faster-whisper` translation library (utilizing the `CTranslate2` engine) to run speech-to-text transcription directly on the host CPU. On the frontend side, real-time voice speech-to-text input is handled natively by the browser's Web Speech API using the `webkitSpeechRecognition` constructor.

### In-Depth Image Processing and OCR Pipeline
When users attach images to the knowledge base or chat, the ingestion pipeline combines optical character recognition with high-level visual descriptions. In [extraction.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/services/extraction.py), the image is loaded via the PIL (Pillow) library for reading and format validation, and passed to the `pytesseract` library (a Python wrapper for Google's Tesseract OCR engine) to extract any embedded text paragraphs. Concurrently, the backend converts the image bytes to a base64 string and forwards it to the cloud vision client. It tries the Gemini API (using the `gemini-3.5-flash` model endpoint) first, and falls back to the OpenAI Vision API (using the `gpt-4o-mini` model endpoint) if needed. If cloud keys fail, it falls back to the local Ollama Vision API (using the `llava-phi3:3.8b` multimodal vision model) as a last-resort fallback. The vision model generates a detailed paragraph describing the image layout, subject, and features. The OCR text and visual description are combined into a single text body and routed to the chunking service, ensuring that images are fully indexed and searchable in the vector database.

### In-Depth Web Search and Site Crawling Pipeline
If the vector database search returns no matching local chunks, the system falls back to active web search. In [web_search.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/services/web_search.py), the system first queries the DuckDuckGo search client using the `duckduckgo_search` Python library (instantiated via `DDGS` context manager) to retrieve matching search URLs. It then triggers parallel HTTP scraping tasks for the top URLs using the `Crawl4AI` python crawling library (using `AsyncWebCrawler`) or the external `Firecrawl` API client (`https://api.firecrawl.dev/v1/scrape`). If both crawlers fail, the system falls back to the Google Gemini API (`gemini-3.5-flash` search grounding features) to query the web. The raw HTML contents are cleaned of menu headers, sidebars, and navigation lists, leaving clean markdown text blocks. These scraped blocks are truncated to three thousand characters to avoid blowing out the model's context window. They are then compiled into a structured prompt context, allowing the assistant to answer questions using real-time internet data.

### Embedding Generation and Schema Vector Size Alignment
After raw text extraction, the document is sent to the chunking service, which slices the content into blocks of up to five hundred characters with a small trailing overlap to preserve context. In [embedding.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/services/embedding.py), the system requests dense vector representations of these chunks. The primary provider is set dynamically in [.env](file:///c:/Users/Tarek/Desktop/inquix/.env) (e.g. Gemini). Because the PostgreSQL database expects vector columns of exactly three thousand seventy-two dimensions, the embedding service performs size matching on the returned list. If a provider returns a vector of a different size (such as local nomic-embed-text returning seven hundred sixty-eight dimensions), the service pads the array with zeros to match the schema. This design prevents database execution failures while supporting multiple embedding models.

### Knowledge Base Retrieval and Vector Cosine Search
When a user submits a chat query, the backend converts the query text into a dense vector embedding. In [retrieval.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/services/retrieval.py), the system calls the pgvector extension on PostgreSQL to perform a cosine similarity calculation comparing the query vector against the embeddings of all chunks in the active database. The system retrieves chunks that meet the similarity threshold configured in settings. If the highest similarity score of the matching chunks is greater than or equal to the designated threshold, the system flags the query as a local lookup. The retrieved local paragraphs are assembled into a context prompt to guide the LLM, bypassing web crawling.

### Large Language Model Stream Generation and Fallback Chain
The prompt context and history are routed to [generation.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/services/generation.py). The generator operates as an asynchronous stream to keep token latency low. The system runs a fallback loop that starts with the user's primary choice (like Groq's llama-3.3-70b-versatile). If a query contains image attachments, Groq is bypassed in favor of Gemini because Groq lacks multimodal support. If the primary cloud service fails (such as returning a rate limit error), the fallback chain tries Gemini, OpenAI, and finally local Ollama in sequence. To handle the rate limits on the Gemini free tier, the system is configured to route all Gemini requests to the `gemini-3.5-flash` model, which features active free tier quota.

### Async Event Stream Flushing and Real-Time Typist Effect
Inside the chat view in [views.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/views.py), the server returns a Django StreamingHttpResponse with cache-control headers disabled. The view yields an empty token immediately to open the connection. As tokens arrive from the model, the backend yields them to the client and sleeps for ten milliseconds via asyncio. sleep. This sleep statement yields execution back to Uvicorn, forcing the ASGI socket to flush the write buffer immediately. This prevents the server from batching tokens into large blocks. On the frontend, [ChatInterface.tsx](file:///c:/Users/Tarek/Desktop/inquix/frontend/components/ChatInterface.tsx) reads the incoming chunks and renders the text, showing a smooth, real-time typing effect.

### Frontend Transition Logic and Audio Synthesis
In [ChatInterface.tsx](file:///c:/Users/Tarek/Desktop/inquix/frontend/components/ChatInterface.tsx), the message loop updates the active message state. When the backend completes generation, it yields a completion payload containing the finalized conversation ID. The frontend updates the initial conversation ID prop, but ignores it if it matches the current state. This check prevents the chat pane from flashing or showing a loading spinner. For text-to-speech feedback, [tts.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/services/tts.py) processes the response text using Kokoro or Edge-TTS, streaming audio buffers to the frontend.

### Multi-Layer Redis Caching and Performance Engineering
To ensure low latency across sessions, the system runs Redis caching via [cache.py](file:///c:/Users/Tarek/Desktop/inquix/backend/app/services/cache.py). Text embeddings and web search queries are cached by hashing normalized search payloads. Full LLM generated stream outputs are also cached. When a cache hit occurs, the backend simulates real-time generation by streaming the cached text to the browser in small blocks separated by a five-millisecond delay. This optimization maintains the typing animation for cached answers, while bypassing the computation time and costs of calling external APIs.
